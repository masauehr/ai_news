## 概要

**Model Context Protocol（MCP）** は、2024年11月にAnthropicが発表したオープンスタンダードで、LLM（AIエージェント）と外部ツール・データソースを統一的なプロトコルで接続するための仕様だ。発表からわずか1年半で **公開サーバー1万以上・月間SDKダウンロード9,700万件** を超え、AWS・Salesforce・Stripeをはじめとする大手企業が採用。2025年12月にはLinux Foundation傘下の「Agentic AI Foundation（AAIF）」へ移管され、事実上の業界標準となった。

日本でも freee・GMO ペパボが国内初のエンタープライズ向けMCPサーバーを公開し、国内普及が加速している。一方で **30件超のCVEが60日間で報告された** というセキュリティ事件も起きており、急成長に伴う課題も浮上している。

---

## MCPとは何か

### 背景：「情報サイロ問題」

LLMは知識を内部に持つが、リアルタイムデータ・社内システム・外部APIへのアクセスは各ツールが独自の方法で実装していた。MCPはこの **「毎回カスタムインテグレーションを書かなければならない問題」** を解決するために設計された。

> "AI models are cut off behind information silos and legacy systems."（AIモデルは情報サイロとレガシーシステムの背後に閉じ込められている）— Anthropic, 2024年11月

### アーキテクチャ

MCPは **JSON-RPC 2.0** ベースのクライアント・サーバー型プロトコル。

```
AIエージェント（MCP Client）
    ↕ MCP プロトコル（JSON-RPC 2.0）
MCPサーバー（Tools / Resources / Prompts）
    ↕ 既存API・DB・ファイルシステム
```

3種類のプリミティブを定義：
- **Tools**: 外部サービスへのアクション（API呼び出し、ファイル操作）
- **Resources**: データの読み取りアクセス（ドキュメント、DB）
- **Prompts**: 再利用可能なプロンプトテンプレート

参照: [Anthropic公式発表](https://www.anthropic.com/news/model-context-protocol) / [MCP仕様](https://modelcontextprotocol.io/development/roadmap)

## 2026年ロードマップ

Anthropicが公式ブログで公開した4つの戦略的優先事項：

### 1. トランスポートの進化（Q1〜Q2）

- **Streamable HTTPのステートレス化**: 複数サーバーインスタンス間での水平スケーリングを実現
- **`.well-known` エンドポイント**: サーバーメタデータの自動発見機能
- **目的**: ロードバランサー・プロキシ背後での安定動作を保証

### 2. エージェント間通信（Q3）

- **Tasks プリミティブの成熟**: 状態管理（working → input_required → completed/failed/cancelled）の精密化
- **リトライ・有効期限ポリシー**: 長時間タスクの安定実行を担保
- **用途**: 医療データ分析・コード移行ツール・マルチエージェント協調

### 3. エンタープライズ認証（Q2〜Q4）

- **Q2 2026**: OAuth 2.1 PKCEフロー + SAML/OIDC統合によるSSO対応
- **Q4 2026**: **MCPレジストリ構築**（セキュリティ監査済みサーバーの公式ディレクトリ・SLA付き）
- **仕様リリース予定**: 2026年6月

### 4. ガバナンス成熟（通年）

- Contributor Ladder SEPの策定（参加者→貢献者→リーダー→メンテナーへの進路明確化）
- Working Groupへの自律的な仕様変更提案（SEP）受理権限の委譲

参照: [2026 MCP Roadmap（公式）](http://blog.modelcontextprotocol.io/posts/2026-mcp-roadmap/) / [MCP Enterprise Readiness](https://workos.com/blog/2026-mcp-roadmap-enterprise-readiness)

## 競合・関連プロトコルとの比較

### MCP vs OpenAI Agents SDK

| 観点 | MCP | OpenAI Agents SDK |
|---|---|---|
| 設計思想 | **ベンダー中立**なJSON-RPCプロトコル | OpenAIエコシステム中心のPythonフレームワーク |
| 主な用途 | エージェント ⇔ ツール/データの接続 | エージェントの構築・オーケストレーション |
| ベンダーロック | なし | OpenAI API依存 |
| リリース | 2024年11月 | 2025年3月（Swarmの後継） |
| 関係性 | **補完的**（OpenAI Agents SDKでエージェントを作り、MCPでデータに繋ぐ） | — |

### MCP vs A2A（Agent-to-Agent Protocol）

Googleが主導するA2Aプロトコルは、エージェント間の水平連携を担う。

| 軸 | MCP | A2A |
|---|---|---|
| 接続方向 | **垂直**（Agent ⇔ Tools/Resources） | **水平**（Agent ⇔ Agent） |
| 状態管理 | ステートレス指向（Streamable HTTP） | ステートフル・長時間実行タスク対応 |
| 主導 | Anthropic → Linux Foundation | Google |

**共存パターン**: Google ADK（Agent Development Kit）はv1.0.0以降でMCPをネイティブサポート。データアクセスにMCP・エージェント間調整にA2Aを使う「縦と横の組み合わせ」が実践的アーキテクチャとして定着しつつある。

参照: [MCP vs A2A 解説](https://auth0.com/blog/mcp-vs-a2a/) / [OpenAI Agents SDK vs MCP](https://blog.promptlayer.com/openai-agents-sdk-vs-mcp/)

## 標準化の里程碑：Linux Foundation への移管

### Agentic AI Foundation（AAIF）設立（2025年12月9日）

- **共同設立**: Anthropic・OpenAI・Block（Square/Cash App親会社）
- **参画企業**: AWS・Google・Microsoft・Salesforce・Snowflake・Cloudflare・Bloomberg
- **初期プロジェクト**: MCP・goose（Block）・AGENTS.md（OpenAI）

この移管によりMCPは単一ベンダーの仕様から **業界共有のインフラ標準** へ格上げされた。USBやHTTPのように「誰もが同じプロトコルで繋げる」世界を目指す方向性が明確になった。

参照: [Linux Foundation AAIF設立発表](https://www.linuxfoundation.org/press/linux-foundation-announces-the-formation-of-the-agentic-ai-foundation-aaif) / [Anthropic MCPをLinux Foundationへ寄贈](https://www.anthropic.com/news/donating-the-model-context-protocol-and-establishing-of-the-agentic-ai-foundation)

## セキュリティ：急速な普及の影

### 「30 CVEs in 60日」事件（2026年1〜2月）

MCPエコシステムで2026年1〜2月の60日間に **30件超のCVE** が報告され、**43万7,000件** の侵害されたダウンロードが記録された。

### 代表的な脆弱性

| CVE | 深刻度（CVSS） | 内容 |
|---|---|---|
| CVE-2025-6514（mcp-remote） | **9.6（Critical）** | OAuth discoveryフィールドへのOSコマンド注入。約500万DL実績のパッケージ |
| CVE-2025-54136（MCPoison） | 高 | CursorのMCPサーバー再検証なし→サーバー更新時に悪意ロジックが静かに注入可能 |
| CVE-2026-32211（Azure MCP） | 高 | 認証機構の完全欠落。認可なしで機密データへアクセス可能 |

### 実世界インシデント

- **Supabase（2025年中盤）**: Cursorエージェントが権限昇格状態でユーザー入力を処理 → SQLインジェクション
- **Asana（2025年6月）**: MCPインスタンス間で顧客情報が漏洩するバグ
- **Postmark（2025年9月）**: サプライチェーン攻撃によりAPIキー・環境変数が窃取

### 脆弱性カテゴリ分布

| カテゴリ | 割合 |
|---|---|
| exec/シェルインジェクション | 43% |
| MCPツーリングインフラ | 20% |
| 認証バイパス | 13% |
| パストラバーサル | 10% |

調査では **82%のMCPサーバーにパストラバーサル脆弱性**が確認されており、**38〜41%のサーバーに認証機構が存在しない** という実態も明らかになっている。

### 対策の現状

**仕様レベル**（2025-11-25版以降）:
- OAuth 2.1 PKCEフロー（ブラウザベースエージェント向け）
- SAML/OIDC統合（エンタープライズIDプロバイダー）
- Client ID Metadata Documents（CIMD）によるURLベース登録

**実装ベストプラクティス**:
- Dockerコンテナ化・ネットワークポリシーによるセグメンテーション
- MCPTox・MindGuard等の専用セキュリティツール使用
- デジタル署名・バージョンロック
- ヒューマン・イン・ザ・ループ（高リスク操作への人間の確認フロー）

参照: [MCP Security Vulnerabilities](https://www.practical-devsecops.com/mcp-security-vulnerabilities/) / [30 CVEs in 60 Days](https://www.heyuan110.com/posts/ai/2026-03-10-mcp-security-2026/)

## まとめ

MCPは「単なるAnthropicのプロプライエタリ仕様」から **AIエージェント時代のインフラ標準** へと地位を確立しつつある。Linux Foundation移管・10,000超のサーバー・AWS等の大手採用はその証左だ。日本でも freee・GMOペパボが先行し、エージェントと業務システムを繋ぐ「配管工事」の国内整備が進んでいる。

ただし急成長の影でセキュリティリスクが顕在化しており、「MCPサーバーを使う/作る組織は、セキュリティ対策を後回しにできない」という認識を早期に持つことが重要だ。2026年後半に整備される認証強化・MCPレジストリの動向が、エンタープライズ本番利用への普及速度を左右する。
